First-principles study guide

Canvas Bundle Sync: what the CLI knows, what it cannot know, and why

Sync is a comparison between a local filesystem bundle and the current server canvas. The subtlety is not the comparison itself. The subtlety is remembering enough about the last synchronized state to tell who changed what.

B

Baseline

What the server actor looked like at the last successful pull or refresh. This is the common ancestor needed for a true three-way comparison.

L

Local

The actor assembled from actor.yaml, native code files, its graph edges, and its position in canvas.yaml.

S

Server

The actor exported by the API now, plus its current platform editVersion.

The previous implementation stored the baseline version, but not the baseline content. It has VB, L, S, and VS. It does not have B. That missing value is the source of the sync ambiguity.

What bare pull does now

Both histories enter diff.conflicts. Pull exits with the conflict code before its merge function runs. A clean local bundle cannot fast-forward a server edit.

What a baseline hash would prove

History A has hash(L) == hash(B), so pull can safely take the server actor. History B has both hashes different from the baseline, so pull must abort.

The current recovery loop can deadlock conceptually. Push says “pull or force local.” Bare pull sees the same server-edit and also aborts. Today, explicit server-wins recovery is bundle pull --replace, which rewrites all managed bundle paths.

05 / Previous edge cases

What every current verdict actually means

These are actor-level outcomes. Canvas metadata follows a separate local-versus-server comparison and has no stored metadata baseline.

Observed state Previous verdict Push Pull Nuance
Both exist; canonical content equal unchanged Skip Skip without file churn A server version may have advanced and returned to identical content; equality makes convergence safe.
Both exist; content differs; stored and server edit versions equal local-edit Update with current editVersion Keep local The unchanged server version proves the server did not move after the baseline.
Both exist; content differs; server edit version advanced server-edit Abort unless --force-local Abort Could be server-only or concurrent editing. Edit-version-only state cannot distinguish them.
Both exist; content differs; this actor has no stored version version-missing Abort unless forced Abort Freshness is unknowable. A completely marker-less bundle gets a warned local-wins compatibility push.
Local only; no stored version new-local Add Keep local Treated as intentionally created in the bundle.
Local only; stored version exists deleted-on-server No actor operation Delete local folder Edit-version-only state cannot tell whether the local actor was edited before the server deletion.
Server only; local actor absent server-only Remove from server Write locally Edit-version-only state cannot tell a new server actor from a locally deleted baseline actor.
Preflight succeeds; another writer edits before batch apply in-flight conflict Some operations may apply; later stages stop Not applicable The API checks edit versions per operation inside one batch; conflicts do not currently make the whole batch all-or-nothing.

06 / Implemented three-way sync

Store a fingerprint of the common ancestor

A content hash is enough. The CLI does not need to store a second full actor document.

Implemented visible sync state

sync:
  actors:
    ACTR...:
      editVersion: 7
      contentHash: sha256:4f3c...

Pairing the version and hash under one actor key prevents two parallel maps from drifting apart.

What the hash represents

  • Canonical actor fields from actor.yaml.
  • Externalized native code re-inlined.
  • Outgoing edges and graph position.
  • Exported actor version excluded.

Direction-specific conflicts are essential. A server-only edit is a push conflict because local is stale, but it is not a pull conflict because local is provably untouched. A concurrent edit is a conflict in both directions.

07 / Deletions

Presence is also part of the three-way state

Hashes fix more than server fast-forward. They distinguish intentional deletion from a new actor, and detect edit-versus-delete conflicts.

Baseline Local now Server now Three-way meaning Safe behavior
Absent Present Absent new local Push adds; pull keeps local.
Absent Absent Present new server Pull writes locally; push policy must deliberately decide whether bundle authority removes it.
Present as B Absent Equal to B local deletion Pull preserves absence; push removes with editVersion.
Present as B Absent Different from B delete / server-edit conflict Abort both directions; require explicit choice.
Present as B Equal to B Absent server deletion Pull removes locally; push should not resurrect silently.
Present as B Different from B Absent local-edit / delete conflict Abort both directions; preserve local files until resolved.

Capability API change? Reason
Store canonical actor baseline hashes No The CLI computes and writes them into canvas.yaml.
Fast-forward server-only actor edits on pull No Current export content plus actor edit versions are sufficient once the local baseline hash exists.
Reject invalid actor configuration in strict batch mode Already supported The API accepts ?strict=true, and the CLI now passes it from push --strict.
All-or-nothing actor batch when one editVersion conflicts Recommended Today non-conflicting operations can commit while conflicting operations are skipped.
Optimistic concurrency for canvas metadata Recommended The metadata update currently accepts no expected canvas version.
One atomic export containing content and actor versions Recommended The CLI currently makes two requests. A combined snapshot removes pairing races; a CLI retry fence can mitigate them meanwhile.
push --strict now reaches strict actor batch validation. It makes local bundle warnings fatal and passes the batch API's existing strict=true query flag. It does not separately call canvases validate.

09 / Study path

A compact way to reason about any sync report

Work in this order. It prevents command flags from obscuring the underlying state transition.

Check presence

Does the actor exist in the baseline, local bundle, and current server export?

Check content

Compare canonical local and server forms. Formatting alone should disappear.

Check ancestry

Ask whether local and server each still equal the baseline. The implemented sync.actors hashes answer this per actor.

Choose direction

Push makes the server follow intentional local work. Pull incorporates safe server work without discarding local work.

Practical rule until baseline hashes exist Treat bare pull as fail-closed on any advanced server actor. Use pull --replace only when you explicitly accept a full server-wins rewrite, and use push --force-local only when you explicitly accept local wins.
